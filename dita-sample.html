Structured-Authoring Sample

DITA Structured-Authoring Project

A small DITA map built from concept, task, and reference topics, using content reuse and conditional processing to publish the same source to both HTML and PDF.

Why this sample looks the way it does: the underlying product and company names are fictionalized because my actual structured-authoring work is proprietary. The map structure, topic types, and reuse techniques below reflect real practice from my work at Epiroc Drilling Solutions.
Architecture

The DITA Map

The map file is the table of contents for a structured-authoring project. It doesn't contain content itself. It references individual topic files and organizes them into a sequence, so the same topics can be reordered or reused in a different map without editing the topics.

hydraulic-maintenance.ditamap 
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE map PUBLIC "-//OASIS//DTD DITA Map//EN" "map.dtd">
<map title="Atlas 900 Series - Hydraulic System Maintenance">
  <topicref href="concepts/hydraulic-system-overview.dita" />
  <topicref href="tasks/replace-hydraulic-filter.dita" />
  <topicref href="tasks/bleed-hydraulic-lines.dita" />
  <topicref href="reference/hydraulic-fluid-specs.dita" />
</map>
Concept

Concept Topic

Concept topics explain what something is before a task topic explains how to work on it. Keeping this separate from procedural content means the same explanation can be linked from multiple tasks instead of repeated in each one.

concepts/hydraulic-system-overview.dita 
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE concept PUBLIC "-//OASIS//DTD DITA Concept//EN" "concept.dtd">
<concept id="hydraulic-overview">
  <title>Hydraulic System Overview</title>
  <conbody>
    <p>The hydraulic system powers the rig's rotation, feed, and clamping
    functions using a closed-loop fluid circuit rated for continuous
    duty operation.</p>
  </conbody>
</concept>
Task

Task Topic

Task topics are written as a numbered sequence of actions, with each step limited to one action so a technician can follow along without reinterpreting the instruction.

tasks/replace-hydraulic-filter.dita 
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE task PUBLIC "-//OASIS//DTD DITA Task//EN" "task.dtd">
<task id="replace-filter">
  <title>Replace the Hydraulic Filter</title>
  <taskbody>
    <steps>
      <step><cmd>Relieve system pressure before opening the filter housing.</cmd></step>
      <step><cmd>Remove the housing cap and extract the used filter element.</cmd></step>
      <step><cmd>Insert the replacement element and reseal the housing.</cmd></step>
    </steps>
  </taskbody>
</task>
Reference

Reference Topic

Reference topics hold lookup information, specs, parameters, or settings, formatted as tables so a reader can scan for a value instead of reading prose to find it.

reference/hydraulic-fluid-specs.dita 
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE reference PUBLIC "-//OASIS//DTD DITA Reference//EN" "reference.dtd">
<reference id="fluid-specs">
  <title>Hydraulic Fluid Specifications</title>
  <refbody>
    <simpletable>
      <sthead><stentry>Property</stentry><stentry>Specification</stentry></sthead>
      <strow><stentry>Viscosity grade</stentry><stentry>ISO VG 46</stentry></strow>
      <strow><stentry>Operating temperature range</stentry><stentry>-20°C to 80°C</stentry></strow>
    </simpletable>
  </refbody>
</reference>
Reuse & Conditions

Content Reuse and Conditional Processing

A shared warning stored once can be pulled into any task that needs it using a conref, so updating the warning in one place updates it everywhere it's referenced.

<caution conref="shared-content.dita#safety-warnings/pressure-relief-warning"/>

Profiling attributes let one source topic serve more than one audience. The build process includes or excludes content based on the attribute, so field technicians and factory technicians can read the same topic with different instructions where the procedure actually differs.

<p audience="fieldTech">Torque the housing cap to 35 Nm.</p>
<p audience="factoryTech">Torque the housing cap to 40 Nm per the rebuild specification.</p>
Publishing

Publishing to HTML and PDF

The map and topics above are the single source of content. The same files publish to an HTML help system for field technicians using tablets, and to a PDF technical manual for printed reference in the shop. Content only needs to be written and maintained once, and both outputs stay consistent with each other because they're generated from the same source rather than kept as separate documents.
